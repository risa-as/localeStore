# روابط Prime API المستخدمة — الغرض والنتيجة والفائدة

توثيق لكل نقطة (endpoint) في API برايم استخدمتها أثناء بناء التكامل: **ماذا تفعل**،
**ما نتيجتها عندنا**، و**هل هي مُستخدمة في الكود الحيّ أم للاستكشاف/التحقق فقط**.

## العناوين (Base URLs)

| البيئة | الرابط |
|---|---|
| الإنتاج | `https://prime-iq.com/myp/webapi` |
| الاختبار (dev) | `https://devtest.prime-iq.com/myp/webapi` |
| توثيق Scalar | `https://devtest.prime-iq.com/myp/scalar/external` |
| ملف المواصفات | `…/webapi/openapi.json` (الكل) · `?scope=external` (نقاط التكامل فقط) |

> كل النقاط المحمية تحتاج ترويسة `Authorization: Bearer <accessToken>`.

---

## 1) المصادقة

### `POST /auth/external-system-login`
- **الغرض:** تسجيل دخول نظام التكامل والحصول على `accessToken` المطلوب لكل نداء آخر.
- **الجسم:** `{ login, password, initialToken }`.
- **النتيجة عندنا:** `200` + `{ accessToken, refreshToken, requiresChoice:false }`. حساب `JOYSTORE` يعمل على **الإنتاج** فقط (يفشل 401 على dev).
- **الفائدة:** أساس التكامل — منه نأخذ التوكن. **مُستخدم في الكود الحيّ** (`prime.ts` → `login()` مع كاش 50 دقيقة).

---

## 2) عمليات البحث العامة (Lookups)

### `GET /general/states`
- **الغرض:** قائمة المحافظات (الولايات) برموزها.
- **النتيجة:** 18 محافظة بصيغة `{ key:"BGD", val:"بغداد" }`.
- **الفائدة:** بناء `PRIME_STATE_MAP` (اسم المحافظة → رمز الولاية المطلوب في الشحنة). استخدمته **مرة واحدة** لملء الخريطة في الكود.

### `GET /external/list-of-system-steps`
- **الغرض:** قائمة حالات/خطوات برايم التي يجب ربطها بنظامنا.
- **النتيجة:** 34 خطوة بصيغة `{ id, code:"DLEIVERD", name, nameEn }`.
- **الفائدة:** بناء `PRIME_STATUS_MAP` (رمز حالة برايم → حالة نظامنا: completed/returned/failed…). استخدمته **مرة واحدة** لملء الخريطة.

### `GET /external/list-of-system-rtn-reasons`
- **الغرض:** أسباب الإرجاع/التعذّر لدى مندوبي التوصيل.
- **النتيجة:** قائمة أسباب (`CLS`, `DENIED`, `NO_ANSWER`…).
- **الفائدة:** مرجعي فقط (للاطّلاع على معاني التعذّر). **غير مُستخدم في الكود حالياً**.

### `GET /general/state-districts/{state-code}` (مرجعي)
- **الغرض:** قائمة المناطق داخل محافظة.
- **الحالة:** لم نعتمده — نرسل `district:"0"` افتراضياً. متاح للتوسعة مستقبلاً.

---

## 3) التجّار والمتاجر (تشخيص الربط + اكتشاف senderId)

### `GET /external/merchants`
- **الغرض:** التجّار **التابعون لحساب التكامل** الحالي.
- **النتيجة عندنا:** `[]` لحساب JOYSTORE (صفر تجّار).
- **الفائدة:** **مِجَسّ الربط الأهم** — أثبت أن لا تاجر مربوط بـ JOYSTORE، فاستنتجت أن `taha1133` غير مربوط.

### `GET /external/merchant-shops/{merchant-login-id}`
- **الغرض:** متاجر تاجر معيّن — كل `shop id` هو قيمة `senderId` المستخدمة في الشحنات.
- **النتيجة:** على dev (تاجر اختبار) رجع المتجر `id:86596` ✅. على الإنتاج `taha1133` رجع خطأ **30**.
- **الفائدة:** مصدر `senderId`، **والتحقق من الربط**. الخطأ 30 = «التاجر غير تابع لنظام التكامل الحالي» (تعريف رسمي).

### `POST /external/create-merchant` (اختبار على dev فقط)
- **الغرض:** إنشاء تاجر تحت نظام التكامل (للـ aggregators).
- **النتيجة:** على dev نجح بعد إضافة الحقول المطلوبة (`name, loginId, password, phone1, branch, state, district`) ورجع تاجراً بمعرّف.
- **الفائدة:** أثبت أننا **نستطيع التسجيل الذاتي للتاجر** عبر API عند الحاجة. **لم يُنفَّذ على الإنتاج** (تجنّباً للازدواجية مع متجرك الموجود).

### `POST /external/create-merchant-shop/{merchant-login-id}` (اختبار على dev فقط)
- **الغرض:** إنشاء متجر لتاجر → يولّد `senderId`.
- **النتيجة:** على dev نجح ورجع متجراً `id:86596`، ثم ظهر في `merchant-shops`.
- **الفائدة:** أكمل إثبات سلسلة التسجيل الذاتي (تاجر → متجر → senderId).

## 4) الشحنات (جوهر التكامل)

### `POST /external/create-shipments`
- **الغرض:** إنشاء شحنة (أو أكثر) في برايم. الجسم **مصفوفة** شحنات.
- **النتيجة:** على dev رجع **خريطة** `{ "23247433": 23247433 }` — القيمة هي **رقم الشحنة**. نخزّنه كـ `deliveryTrackingId`.
- **الفائدة:** **أهم نقطة** — تُرسِل الطلب لبرايم. **مُستخدمة في الكود الحيّ** (`sendOrder`) عند تحويل الطلب إلى «معلّق».

#### عناصر كل شحنة (الحقول)

الجسم **مصفوفة** كائنات. كل كائن:

**🔴 إجبارية (يرفض برايم الطلب بدونها):**

| الحقل | النوع | المعنى | ماذا يرسل نظامك | كود الخطأ عند نقصه |
|---|---|---|---|---|
| `receiverName` | string | اسم المستلم | اسم الزبون | **4007** |
| `receiverHp1` | string | هاتف المستلم (محلي `07…`) | هاتف الزبون | **4009** ناقص / **29** خاطئ |
| `receiptAmtIqd` | integer | المبلغ المُحصَّل (**دينار كامل**) | سعر الطلب | **4010** (يجب > 0) |
| `senderId` | integer | رقم متجرك | `PRIME_SENDER_ID` | **4000** المتجر غير متوفر |
| `merchantLoginId` | string | معرّف التاجر | `PRIME_MERCHANT_LOGIN_ID` | مرتبط بـ 4000/30 |
| `state` | string | رمز الولاية (`BGD`…) | `PRIME_STATE_MAP[المحافظة]` | — |
| `qty` | integer | عدد القطع | كمية الطلب | — |

**🟡 شبه إجبارية (يُفضّل إرسالها):**

| الحقل | النوع | المعنى | ماذا يرسل نظامك |
|---|---|---|---|
| `productInfo` | string | نوع/وصف البضاعة | أسماء المنتجات |
| `locationDetails` | string | تفاصيل العنوان | العنوان (+ الملاحظات مدموجة) |

**⚪ اختيارية / لها قيم افتراضية:**

| الحقل | النوع | المعنى | قيمة نظامك الافتراضية |
|---|---|---|---|
| `district` | string | رمز المنطقة | `"0"` |
| `haveReturnItems` | string | يحتوي إرجاع؟ (`Y`/`N`) | `"N"` |
| `custReceiptNoOri` | integer | رقم الوصل (يولّده برايم إن `0`) | `0` |
| `senderSystemCaseIdWithCharacters` | string | **مرجعك الفريد** للطلب | معرّف الطلب عندك |

**مثال طلب كامل:**

```json
[
  {
    "receiverName": "احمد خالد",
    "receiverHp1": "07701234567",
    "receiptAmtIqd": 30000,
    "senderId": 125977,
    "merchantLoginId": "taha1133",
    "state": "BGD",
    "qty": 1,
    "productInfo": "تيشيرت + بنطال",
    "locationDetails": "المنصور - قرب الحديقة — توصيل صباحاً",
    "district": "0",
    "haveReturnItems": "N",
    "custReceiptNoOri": 0,
    "senderSystemCaseIdWithCharacters": "a1b2c3d4"
  }
]
```

> ملاحظات: الهاتف محلي `07…` (لا `+964`)؛ المبلغ كامل (`30000` لا `30`) وأكبر من صفر.
> في نظامك يُبنى هذا تلقائياً في `prime.ts` من بيانات الطلب — لا يُكتب يدوياً.

### `POST /external/shipments-info`
- **الغرض:** تفاصيل شحنات بأرقامها. الجسم **مصفوفة أرقام** `[id,…]`.
- **النتيجة:** مصفوفة فيها `caseid`، `stepCode`/`shipmentStatus` (الحالة)، `receiptAmtIqd` (المبلغ)، `receiverName/Hp1`.
- **الفائدة:** **مصدر المزامنة** — نمرّر أرقام طلباتنا المخزّنة فنعرف حالة كل شحنة. **مُستخدمة في الكود الحيّ** (`fetchRemoteOrders`/`syncModonOrders`). أفضل من البحث لأننا نستعلم بأرقامنا مباشرة.

### `POST /external/search-shipments` (لم نعتمده)
- **الغرض:** بحث بالشحنات بفلاتر تاريخ (صفحات size=10).
- **النتيجة:** رجع `[]` (لا يفهرس فوراً / يحتاج فلاتر دقيقة).
- **الفائدة:** بديل محتمل للمزامنة — استبدلناه بـ `shipments-info` لأنه أدقّ وأسرع لحالتنا.

### `POST /external/calculate-shipment-charges` (تحقق غير مُتلِف)
- **الغرض:** حساب أجور شحنة **دون إنشائها**.
- **النتيجة:** بـ `senderId=125977` رجع خطأ **30** (غير مربوط).
- **الفائدة:** **تحقق آمن** من صلاحية المتجر دون إنشاء شحنة حقيقية على الإنتاج.

### `POST /external/update-case-status` (مرجعي — «request resend»)
- **الغرض:** طلب إعادة إرسال شحنة مع تصحيح عنوان المستلم.
- **الحالة:** متاح لميزة «إعادة الإرسال» مستقبلاً. **غير مُستخدم حالياً**.

## 5) المواصفات (OpenAPI)

### `GET …/webapi/openapi.json` (و `?scope=external`)
- **الغرض:** ملف المواصفات الكامل للـ API.
- **النتيجة:** 345 مساراً (الكل) · 34 مساراً (نقاط التكامل).
- **الفائدة:** منه استخرجت **المخططات الدقيقة وأسماء الحقول وأكواد الأخطاء** بدل التخمين — وعليه بنيت/صحّحت التنفيذ (مثل رد الإنشاء `object<string,int>`، وحقول `shipments-info`، ومعنى الخطأ 30).

## خلاصة منهجية التحقق

أعامل كل نقطة كـ«مِجَسّ»، وأفسّر النتيجة عبر **كود الخطأ الرسمي**:
1. `login` ✅ → بيانات التكامل سليمة.
2. `states`/`steps` ✅ → التوكن صالح فعلاً للعمليات.
3. `merchants` = `[]` → لا تاجر تابع للتكامل.
4. `merchant-shops/taha1133` و`calc-charges` → خطأ **30** = «التاجر غير تابع لنظام التكامل الحالي».

⟵ تقاطُع هذه الأدلة أثبت أن العائق الوحيد هو **ربط `taha1133` بـ `JOYSTORE`** عند برايم.
